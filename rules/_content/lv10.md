# Slime Architecture Rules — Level 10 (CQRS)

This project uses **Slime Architecture**, a progressive layered architecture for TypeScript/Node.js backends.
The current level is **Lv10**: the highest level. Read and write concerns are separated via CQRS within each domain.

## CQRS Rules (new at Lv10)

### command/ — write side

- Contains all operations that **change state**: create, update, delete.
- `workflow.ts` orchestrates the write: parse → logic → writePort.
- `ports.ts` defines the `WritePort` interface.
- `logic.ts` contains business judgment functions (Result type) for this domain.
- `logic.test.ts` is mandatory.
- Command workflows should be **optimized for correctness and consistency**, not for read performance.
- Command workflows emit domain events (return them) after successful writes.

```ts
// domainUser/command/ports.ts
export type UserWritePort = {
  save: (user: NewUser) => Promise<User>
  delete: (id: string) => Promise<void>
}

// domainUser/command/workflow.ts
export const createUserWorkflow =
  (readPort: UserReadPort, writePort: UserWritePort) =>
  async (input: ValidatedInput): Promise<UserCreatedEvent> => {
    const exists = await readPort.findByEmail(input.email)  // read to check precondition
    const check = userCanCreate(!!exists)
    if (!check.ok) throw new Error(check.error)
    const user = await writePort.save(input)
    return { type: "USER_CREATED", payload: { userId: user.id, email: user.email } }
  }
```

### query/ — read side

- Contains all operations that **read state without changing it**: get, list, search.
- `workflow.ts` orchestrates the read: parse → readPort → response.
- `ports.ts` defines the `ReadPort` interface.
- No `logic.ts` required unless there is non-trivial read-side computation.
- Query workflows may return **denormalized or view-optimized data structures** — not constrained to the write model shape.
- Query workflows must **never trigger side effects** (no writes, no event emissions).

```ts
// domainUser/query/ports.ts
export type UserReadPort = {
  findByEmail: (email: string) => Promise<User | null>
  findById: (id: string) => Promise<UserView | null>
  listActive: () => Promise<UserView[]>
}

// domainUser/query/workflow.ts
export const getUserWorkflow =
  (readPort: UserReadPort) =>
  async (input: { id: string }): Promise<UserView | null> => {
    return readPort.findById(input.id)
  }
```

### routes.ts — routing both sides

- Routes wire both command and query workflows.
- HTTP verbs naturally map to sides: `GET` → query, `POST`/`PUT`/`PATCH`/`DELETE` → command.

```ts
// domainUser/routes.ts
export const userRoutes = (r: Router) => {
  // Query side
  r.get('/users',     slime.auth(), GetUserListWorkflow)
  r.get('/users/:id', slime.auth(), GetUserWorkflow)
  // Command side
  r.post('/users',      slime.auth(), CreateUserWorkflow)
  r.patch('/users/:id', slime.auth(), UpdateUserWorkflow)
  r.delete('/users/:id', slime.auth(), DeleteUserWorkflow)
}
```

## infrastructure/ Rules (unchanged from Lv9, now split by side)

- Separate adapter implementations for read and write if data access patterns differ.
- Read adapters may use different query optimizations (projections, read replicas, caches).
- Write adapters focus on transactional consistency.

```ts
// infrastructure/user/commandAdapter.ts — transactional writes
export const userCommandAdapter: UserWritePort = {
  save: (user) => prisma.user.upsert({ ... }).then(mapToUser),
  delete: (id) => prisma.user.delete({ where: { id } }).then(() => undefined),
}

// infrastructure/user/queryAdapter.ts — read-optimized
export const userQueryAdapter: UserReadPort = {
  findByEmail: (email) => prisma.user.findUnique({ where: { email } }).then(mapToUser),
  findById: (id) => prisma.user.findUnique({ where: { id }, select: userViewFields }).then(mapToView),
  listActive: () => prisma.user.findMany({ where: { status: 'active' } }).then(rs => rs.map(mapToView)),
}
```

## Cross Folder Rules (unchanged from Lv7–Lv9)

- `cross-[kebab-case]` naming. Domains import from `cross-`, not the reverse.
- Cross folders may have `ports.ts` for their own infrastructure boundaries.
- `cross-` folders specialized for transactional operations.

## Domain Folder Rules (unchanged from Lv6–Lv9, scoped to command/ and query/)

- `domain[A-Z]camelCase` naming. Domains must not import from each other.
- All Lv1–Lv9 layer rules apply (now scoped within `command/` and `query/`).

## shared/events.ts Rules (unchanged from Lv8–Lv9)

- All domain event type definitions in one file.
- Events: `{ type: "UPPER_SNAKE_CASE", payload: { ... } }`.
- Command workflows emit events after successful writes.
- Query workflows never emit events.

## Error Handling

```ts
// slime.config.ts
export default {
  errors: {
    USER_ALREADY_EXISTS: 409,
    UNAUTHORIZED: 403,
    FORBIDDEN: 403,
    OUT_OF_STOCK: 422,
    NOT_FOUND: 404,
    PAYMENT_DECLINED: 402,
  }
}
```

- **Validation failures**: framework maps to HTTP 400.
- **Auth/middleware rejection**: framework maps to 401, 403, or 429.
- **Domain errors**: `throw new Error(code)` → framework maps via error config to 4xx.
- **Unhandled throws**: framework maps to HTTP 500.

## General Principles

- **No runtime magic**: routes are always declared explicitly. No file-based auto-routing.
- **No DI container auto-registration**: import and wire dependencies manually.
- **Static over dynamic**: naming conventions are enforced by Kaachan at lint time, not at runtime.
- **Functional Core, Imperative Shell**: logic = pure core, workflow = imperative shell.
- **Dependency Inversion**: domains define ports; infrastructure implements adapters.
- **CQRS**: reads and writes are separate concerns — different models, different optimizations.
- **Query workflows are side-effect free**: if a GET handler triggers a write, it belongs in command.

---

## What Kaachan Checks at Lv10

- Domain folders contain `command/` and `query/` subfolders.
- `query/workflow.ts` does not import `WritePort` or any write adapter.
- `command/workflow.ts` does not return read-optimized view models (view DTOs belong in query).
- ORM/DB imports are confined to `infrastructure/`.
- `logic.test.ts` missing in `command/` → ❌ Error.
- All Lv9 checks continue to apply.

## Guidance for AI Assistants

When adding a new feature:
1. Determine first: is this a **command** (changes state) or a **query** (reads state)?
2. Place it in the appropriate `command/` or `query/` subfolder.
3. If a command needs to check a precondition (e.g., "does this user exist?"), it may use a `ReadPort` — but only for the precondition check, not to build a response.
4. Never add write operations to a query workflow — refactor into a command workflow instead.
5. Design `ReadPort` methods for query workflows with the view shape in mind — avoid forcing the query side to reconstruct views from normalized write models.

When suggesting a migration from Lv9 to Lv10:
1. Audit existing workflows: which ones only read, which ones only write, which ones do both?
2. Split mixed workflows into separate command and query workflows.
3. Separate the port interfaces: `WritePort` in `command/ports.ts`, `ReadPort` in `query/ports.ts`.
4. Create separate infrastructure adapters for read and write if their data access patterns differ.
5. Update `routes.ts` to wire the split workflows.
