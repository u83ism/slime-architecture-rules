# 👩 Kaachan 設計仕様

> **位置づけ：** 本ドキュメントはSlime FWの静的解析ツールKaachan（とその周辺）の実装詳細を記録する開発用資料。概要は [フレームワーク（案）](./🤤%20僕の考えた最強の次世代Webアプリケーションフレームワーク（案）.md) を参照。

## Fat Logic防止戦略

Logic層が最初から持つ最大のリスクは、**ドメイン分割（Lv6）前に際限なく肥大化すること**だ。「行数だけ監視してちゃんとドメイン切れと言う」だけでは機能しない。4つのアプローチを組み合わせて対処する。

### ① ドメインprefixによる命名規則の強制

Logic関数には**ドメインを示すprefixを必須**とする。`userCan*`・`orderCan*`・`paymentCan*` のように。Lv4でStore関数に `find*`/`create*` を要求したのと同じ発想の拡張で、命名するだけでドメイン境界の「下書き」が生まれる。Kaachanが違反を静的解析で検出する。

### ② `logic/` フォルダへの中間昇格

Lv6のドメイン分割は「store・workflow・logicをセットで切り出す」大仕事だ。その前に**ロジックだけ先に整理する中間ステップ**として `logic/` フォルダへの昇格を許可する。`logic/user.ts`・`logic/order.ts` のように関心事ごとに分割でき、これが将来のドメイン分割の雛形になる。Lv6移行の敷居が高くて踏み出せない状況の緩和にもなる。

### ③ 型依存グラフによる関心事分離の自動検出

各Logic関数が「どの型を参照しているか」はts-morphによるAST解析で取得できる。**互いに型依存が一切交差しない関数群は、異なるドメイン候補として機械的に検出できる。** Kaachanが検出して提示する：

```
⚠️ logic.tsで型依存が交差しないグループを検出しました
  Group A: userCanCreate, userCanUpdateProfile（User型系）
  Group B: orderCanPlace, orderCanShip（Order/Item型系）
  → logic/user.ts と logic/order.ts への分割を検討してください（prefix命名とも一致）
```

### ④ Kaachan と rulesファイルの役割分担

| 役割 | 担当 |
|---|---|
| 行数・prefix混在・型依存の非交差を検出（客観的事実） | Kaachan（静的解析） |
| 「なぜ分けるか」の文脈理解と具体的な分割案 | rulesファイル経由のAI |
| 「どこで切るか」の最終判断 | ユーザー ＋ AI との対話 |

Kaachanが「何が起きているか」を検出し、rulesファイルが「どうすべきか」をAIに伝える。静的解析とAIの役割を明確に分担することで、どちらか単独では担えない部分を補い合う。

### エスカレーション段階（3段階）

| 段階 | 条件 | 種別 |
|---|---|---|
| 💡 Hint | 300行 / 関数10個 超過 | 情報提供 |
| ⚠️ Warning | 500行超過 または prefix混在 または 型依存グラフで非交差グループ検出 | 構造的問題あり |
| ❌ Error | `logic/`内の各ファイルが300行超 または prefix混在が未解消 | 移行を阻止 |

## Fat Parse問題

Fat Logicと同じ構造的リスクが `parse.ts` にも存在する。mpywさんの[なんちゃってクリーンアーキテクチャ記事](https://zenn.dev/yumemi_inc/articles/ce7d09eb6d8117)で「FormRequestはcreate用とupdate用で別ファイルにする（共通化しない）」と指摘されていたように、parse.tsも肥大化しやすい。特にLv6以降でドメインが増えると「createUser・updateUser・listUsers・placeOrder…」が1ファイルに混在する問題が起きる。

`logic.ts` → `logic/` の中間昇格パターンと同じ対処が有効：

```
parse.ts      # 初期（閾値超過でparse/へ昇格を促される）
parse/
  createUser.ts
  updateUser.ts
  listUsers.ts
  placeOrder.ts
```

**Logic との差異：** parseは「同一スキーマを複数エンドポイントで使う」ケースが正当に存在するため、「共通化禁止」は課さない（共通化を許容したまま肥大化だけ検知する）。Kaachanの閾値・エスカレーション段階はlogicと同方針でよい。

## エラーHTTP境界マッピング

**DomainError / TechnicalError の二分類**を採用する（mpywさんの[進化的アーキテクチャ記事](https://zenn.dev/yumemi_inc/articles/ebc6e634fa57a1)の例外設計から着想）。

| 分類 | 意味 | HTTPコード |
|---|---|---|
| **DomainError** | LogicのResult err / 業務的な拒否（ビジネスルール違反） | 4xx |
| **TechnicalError** | 未捕捉throw / DB障害等（技術的障害） | 500 |

`slime.config.ts` にDomainErrorの文字列→HTTPコードのマップを一元定義。マップに登録されていないthrowはすべてTechnicalError（500）として扱う：

```ts
// slime.config.ts
export default {
  errors: {
    // DomainError = LogicのResult errで使う文字列 → 4xx
    USER_ALREADY_EXISTS: 409,
    UNAUTHORIZED: 403,
    OUT_OF_STOCK: 422,
    NOT_FOUND: 404,
    // 上記マップにないthrow → 自動的に500（TechnicalError）
  }
}
```

Slimeが自動カバーする範囲：Parse失敗→400 / Middleware拒否→401/403/429 / 未捕捉throw→500。ユーザーがマップに登録するのはLogicのResult errの文字列のみ。LogicはHTTPを一切知らなくて済む。

**導入タイミング：** Lv5（Logic導入と同時）。Kaachanは `new Error()` 直throwに対してwarningを出し、Result err + エラー文字列のマップ登録を促す。

## 依存方向静的検査

deptrac（PHP）や`dependency-cruiser`（JS/TS）相当の**ドメイン間・層間の依存方向チェック**をKaachan内で実行する設計。Lv6以降で有効になる。

### 検査対象ルール（レベル連動）

Kaachanが既に持つ制約ルールをimportグラフ解析で自動検証する：

| ルール | 検出方法 | 適用Lv |
|---|---|---|
| domain間の相互参照禁止 | importグラフ解析 | Lv6〜 |
| Appからのみドメイン参照可能 | importグラフ解析 | Lv6〜 |
| WorkflowがORMを直接importしない | importグラフ解析 | Lv4〜 |
| infrastructureフォルダ外のORM直接操作禁止 | importグラフ解析 | Lv9〜 |

依存ルールはアーキテクチャレベルに連動して自動設定されるため、`slime migrate --to-level 7` でLvが上がると検査対象も自動で増える。

### 実装方針

`dependency-cruiser`を組み込む（外部依存として追加）か、Kaachan内で独自実装するかは要検討。

| 方針 | 利点 |
|---|---|
| `dependency-cruiser`を組み込む | ルール定義の標準化・既存のecoシステムと互換 |
| Kaachan内で独自実装 | Slime固有のルール体系（レベル連動）との統合が容易 |

現時点では`dependency-cruiser`を組み込む方向が有力。Slime固有のレベル連動ルールは`dependency-cruiser`の設定ファイルをSlimeが動的生成する形で対応できる。

## rulesファイル詳細設計

静的解析の限界として触れた「ドメインをどう切るか」「マイグレーションをどうやってするか」という問題に対し、rulesファイル経由で間接的に介入できる可能性がある。

**「FWからAIを呼ぶな」方針との整合性：** FWがAIをAPI呼び出しするのではなく、**AIが動く前提条件（rulesファイル）をFWが整備する**という構造であり、起動者はあくまでユーザー。矛盾しない。

### Lv5のrulesファイル例

```markdown
# Generated by slime export:rules (Level: 5)

## ドメイン分割の提案ルール
workflow.ts または logic.ts が 300行 / 関数10個 を超えている場合：
1. ユーザーにドメインモデリングを提案する
2. ビジネス上の関心事のまとまりをユーザーと一緒に言語化する
3. ユーザーの承認を得た上でフォルダ構成案を提示する
4. Lv6移行を示唆する（`slime level:next` で構造確認）

## Slime Lv5 制約
- Logic関数はドメインprefixを付けること（userCan*, orderCan* など）
- Logic関数はResult型で返すこと（throwしない）
- Logic関数はテストを書くこと
- parseはDB接続不可
```

### KaachanとrulesファイルのHintの閾値を同期する

**KaachanのHintとrulesファイルの閾値を同じ数値に保つ**ことが重要で、「Kaachanが警告を出す」→「AIが提案を続ける」という一貫したメッセージをユーザーが受け取れる。

### rulesファイルのフォーマット問題

現時点でrulesファイルのフォーマットは乱立しており（`.cursor/rules`・`CLAUDE.md`・`.clinerules`・`GEMINI.md`等）、対応ツールを引数で指定するか全フォーマット一括生成とするかも含め、実装方針は未定。

```bash
slime export:rules --target cursor   # .cursor/rulesのみ生成
slime export:rules --target claude   # CLAUDE.mdのみ生成
slime export:rules                   # 全フォーマット一括生成（デフォルト候補）
```
