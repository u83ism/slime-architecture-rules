> **位置づけ：** Slimeのルーティング設計方針。「ファイルベースルーティング vs 明示的ルートファイル」の採否根拠と、Fat Routing問題への対処をまとめる。アーキテクチャ文書の残課題「Fat Parse問題」と同列の問題として、本ドキュメントで設計方針を確定させる。

---

## 現状の空白

アーキテクチャ文書のLv6以降のディレクトリ図を見ると、`app/route.ts` は1ファイルのまま維持されている。Lv1から変わらない。

これはドメイン分割（Logic/Store/Workflow）と対称的ではなく、大規模アプリケーションでは実際に問題になる。極端な例として「web.php・api.php に加えてドメイン別に5つのルーティングファイルが乱立し、合計3000近いルートがある」Laravelアプリが現実に存在する。

Slimeが「スムーズなアップスケーリング」を掲げる以上、ルーティングの肥大化にも設計方針が必要だ。

---

## Fat Routingの解剖

「ルートが3000ある」は複数の問題が混在した症状だ。処方が変わるため分類する。

| 問題 | 例 | 本質 |
|---|---|---|
| **Workflowが本当に多い** | 100リソース × 5動詞 = 500 | 規模の問題。構造的には正常 |
| **バージョン増殖** | `/v1/`, `/v2/`, `/v3/` が全部残存 | 廃止管理の失敗 |
| **別パス同Workflow** | `/api/users` と `/admin/users` が同じ処理 | URLとWorkflowが1:1でない |
| **死んだルート** | Workflowを削除してもルート定義が残存 | 棚卸されていない |

LaravelのFatControllerと同じ構造で、ルートファイルをどう分割してもWorkflowが整理されない限り根本は解決しない。**ルート数よりWorkflow数が本質。**

## ファイルベースルーティングの検討と不採用

Next.jsやSvelteKitが採用するファイルベースルーティングは「先祖返り」という評もあるが実際の系譜はこうだ：

```
PHP時代:   users.php → /users （ファイル = ルート・メソッド問わず全部）
MVC時代:   web.php に全部書く（明示的集中管理）
Next.js:   app/users/page.tsx → /users （先祖返り、ただしフロントエンド特化）
```

Next.jsが先祖返りで成功した理由は**フロントエンドだから**だ。URLとコンポーネントは概ね1:1に対応する。APIサーバーとしてのSlimeはこの前提が成り立たない。

### 採用しない理由

**理由1: URLとドメイン構造が一致しない**

```
/v1/users/:id   ─┐
/v2/users/:id   ─┤→ 同じ GetUserWorkflow を呼ぶ
/api/users/:id  ─┘

/admin/users/:id → AdminGetUserWorkflow
```

ファイルパスで表現すると `routes/v1/users/[id].ts`・`routes/v2/users/[id].ts` がそれぞれ `GetUserWorkflow` をimportするだけのファイルになる。意味のない間接層が増える。

**理由2: ミドルウェア付与が必ず魔法になる**

Next.jsの `middleware.ts`（特定フォルダ以下に自動適用）がその典型。Slimeは `slime.auth()` をルート定義に明示するのが設計の核にあり、「フォルダに設定ファイルを置けば自動適用」という方式は命名規則問題と同じ「ランタイム/規約による暗黙解決」に陥る。

```ts
// Slimeの明示的なミドルウェア付与（これを維持したい）
route.get('/admin/users', slime.auth({ role: 'admin' }), GetAdminUsersWorkflow)
route.get('/users/:id',   slime.auth(),                  GetUserWorkflow)
route.post('/webhook',    verifyStripeSignature,          StripeWebhookWorkflow)
```

**理由3: 同一URLへの複数メソッドが不自然**

```
GET  /users → ListUsersWorkflow
POST /users → CreateUserWorkflow
```

Next.jsはこの問題を `route.ts` 内で `export function GET()` / `export function POST()` と書くことで解決したが、これは**明示的なルートファイルに先祖返り**している。結局ファイルベースでもルートファイルは残る。

### ファイルベースが唯一勝る点

**死んだルートが構造的に存在できない。** ファイルを消せばルートが消える。

Laravelの3000ルート問題の一因は「Controllerを削除してもweb.phpのルート定義が残る」という分離コストだ。これはファイルベースでなくても解決できる（後述）。

## Slimeの解答：domain co-location + Kaachan

### Lv6以降のroute co-location

Logic/Store/Workflowがドメインに同居するのと同じ発想でルートもドメインに同居させる。

```
app/
  route.ts         ← 集約のみ（処理ゼロ）
  workflow.ts
  parse.ts
  middleware.ts

domain/user/
  workflow.ts
  store.ts
  logic.ts
  routes.ts        ← Lv6以降: userドメインのルートをここに

domain/order/
  workflow.ts
  store.ts
  logic.ts
  routes.ts        ← orderドメインのルートをここに
```

```ts
// app/route.ts（集約のみ。URLもミドルウェアも書かない）
import { userRoutes } from '../domain/user/routes'
import { orderRoutes } from '../domain/order/routes'

export const routes = [userRoutes, orderRoutes]
```

```ts
// domain/user/routes.ts（userドメインのルート定義）
route.get('/users',     slime.auth(), ListUsersWorkflow)
route.get('/users/:id', slime.auth(), GetUserWorkflow)
route.post('/users',    slime.auth(), CreateUserWorkflow)
```

**`app/route.ts` が肥大化する = ドメイン分割が不十分**というシグナルとして扱える。KaachanがLogicと同じエスカレーション方式で検知する。

### Lv1-5の扱い

Lv1-5は `app/route.ts` 1ファイルのままで問題ない。Workflowが少なければルートも少ない。ルートファイルの分割はドメイン分割（Lv6）と同タイミングで行う。

## Kaachanへの追加検知

現状の設計に以下が抜けている。Fat Parse問題と同じエスカレーション方針で追加する。

### route.tsの肥大化検知

| 閾値 | 検知 |
|---|---|
| `app/route.ts` が一定規模超過 | 💡 Hint「ルートファイルが大きくなっています。Lv6のdomain co-locationを検討してください」 |
| Lv6以降で `app/route.ts` にルート定義が残存 | ⚠️ Warning（domain/*/routes.tsへの移動を促す） |
| 同一ドメインのルートが複数ファイルに散在 | ⚠️ Warning |

### 孤立ルート・孤立Workflowの検知

ファイルベースルーティングの「ファイルを消せばルートが消える」という利点を、静的解析で代替する。

| パターン | 検知 |
|---|---|
| ルート定義に指定されたWorkflowファイルが存在しない | ⚠️ Warning（死んだルート候補） |
| Workflowファイルは存在するがどのルートからも参照されていない | 💡 Hint（到達不能Workflow候補） |

完全な「死んだルート」検知はランタイムのアクセスログなしには確定できないが、静的解析でファイルの存否チェックは可能。`slime export:openapi` と組み合わせてCIで `slime check:routes` を走らせることで棚卸しを定期化できる。

## api/ 名前空間の確保

### なぜ初日から確保するか

「現在はAPIサーバーのみ」であっても、将来LaravelやRailsのようにwebルートを抱える可能性があるなら、`api/` 名前空間は**初日に確保するコストが最も安い**。

後から分けようとすると全クライアントへの破壊的変更になる：

```
# 移行前（後から全部変わる）
GET /users      → フロント・モバイル・外部連携が叩いているURL

# 移行後（全クライアントを更新しないといけない）
GET /api/users
GET /users      → HTMLページ
```

`api/` の確保は名前空間の**予約**であり、webルートを実際に追加するまで `web/` 空間は空のまま放置してよい。

### route.group による名前空間の確立

```ts
// Lv1-5: route.tsでグループ化
route.group({ prefix: '/api' }, (r) => {
  r.post('/users',    CreateUserWorkflow)
  r.get('/users/:id', slime.auth(), GetUserWorkflow)
})

// /health, /webhooks 等はグループ外（意図的に /api/ なし）
route.get('/health', HealthCheckWorkflow)
route.post('/webhooks/stripe', verifyStripe, StripeWebhookWorkflow)
```

### v1 はオプトイン

`api/` と `v1/` は性質が異なる別の関心なので分けて扱う：

| | `api/` | `v1/` |
|---|---|---|
| 目的 | web/api 名前空間の分離 | APIバージョニング |
| 必要なタイミング | webルートを将来追加するなら初日から | 外部公開・破壊的変更が発生してから |
| Slimeの立場 | **デフォルト推奨** | **オプトイン** |

`v1` が必要になったらグループをネストする：

```ts
route.group({ prefix: '/api' }, (r) => {
  r.group({ prefix: '/v1' }, (r) => {
    r.get('/users', ListUsersWorkflow)
  })
  // /v1 と /v2 を共存させる場合
  r.group({ prefix: '/v2' }, (r) => {
    r.get('/users', ListUsersV2Workflow)
  })
})
```

### Lv6以降のファイル構造

Lv6でdomain co-locationと同時に、`app/` 内でも名前空間をファイルとして分離する：

```
app/
  route.ts         # エントリーポイント（domainルートをimportして束ねる）
  parse.ts
  middleware.ts
  workflow.ts
```

```ts
// app/route.ts
import { userRoutes } from '../domainUser/routes'
import { orderRoutes } from '../domainOrder/routes'
export const routes = [userRoutes, orderRoutes]

// domainUser/routes.ts
export const userRoutes = (r: Router) => {
  r.get('/users',     slime.auth(), ListUsersWorkflow)
  r.post('/users',    slime.auth(), CreateUserWorkflow)
  r.get('/users/:id', slime.auth(), GetUserWorkflow)
}
```

## APIバージョニングとの連携

バージョン増殖問題（`/v1/`, `/v2/` が全部残存）はFW文書のAPIバージョニング機能と連携する。

```ts
// バージョン廃止予告をルート定義の属性として指定
route.get('/v1/users', GetUsersWorkflow, {
  deprecated: true,
  sunset: '2026-09-01'  // Sunset ヘッダーを自動付与
})
```

Kaachanが `sunset` 日付を過ぎたルートを ⚠️ Warning として検知することで、バージョン廃止の棚卸しを機械的に強制できる。
