> **位置づけ：** 「命名規則に従えば自動で動く」系の動作をSlime/Kaachanがどう扱うかの設計方針。Laravelのリフレクション問題を反面教師として、**体験は提供するが仕組みは静的に**という原則を定める。

---

## 問題提起：Laravelのリフレクション問題

Laravelは「命名規則に従えば自動で動く」DXを多くの箇所で提供している。

```
app/Models/User.php    → Eloquentが自動でusersテーブルに紐づける
app/Http/Controllers/  → RouteServiceProviderが自動検出
__construct(UserRepo)  → IoC Containerが自動解決
```

体験としては優れているが、内部実装はランタイムリフレクションに依存している。この結果：

- **コードを読んでも実行フローが追えない**（IDEの「定義に移動」が機能しない）
- **エラーが実行時まで検出できない**（タイポがある命名規則は動かして初めて気づく）
- **静的解析が効かない**（TypeScriptの型安全が意味をなさない）

Laravel固有の問題ではなく、**「命名規則の解決をランタイムに委ねる」設計全般に共通する問題**。

---

## 「命名規則→自動動作」が解決したい本質

この仕組みが解決しようとしている問題は正当だ：

| 目的 | 例 |
|---|---|
| ボイラープレートを減らしたい | ControllerをいちいちDIコンテナに登録したくない |
| 設定より規約でDXを上げたい | 「ここに置いたら動く」というメンタルモデルを提供したい |
| 肥大化・設計崩壊を防ぎたい | 規則を守っているかどうかを機械的に担保したい |

問題は目的ではなく**解決手段**にある。ランタイムリフレクションは「規則に従っているかどうか」を動的に解決するが、この判断は静的にできる。

---

## Slime/Kaachanの解答：規約はある、解決は静的に

Slime/Kaachanの方針：

> **「命名規則に従えば自動で動く」体験は提供する。ただしその解決はランタイムではなく静的に行う。**

### 1. Kaachanによる静的検査（命名規則の静的エンフォース）

Lv4のStore命名規則を例に取る：

```ts
// OK: 命名規則に従っている
export const findUserById = (id: string) => ...
export const saveUser = (user: User) => ...

// Kaachanが検知する NG
export const getUserById = (id: string) => ...   // 💡 Hint: find*を使ってください
export const insertUser = (user: User) => ...    // 💡 Hint: save*を使ってください
```

Laravelは「命名規則に従っていれば（ランタイムに）自動でDBに繋がる」。
Kaachanは「命名規則に従っているかを（Lintタイムに）静的に検査する」。

**実行時には何の魔法も起きない。コードに書いた通りに動く。** 規則の強制はLintタイムで完結する。

Lv4でHintだった命名規則チェックがLv6でErrorになるのも同じ発想——レベルが上がるにつれ規約の強制度が上がるが、それを担うのは常にKaachanの静的解析であり、ランタイムの暗黙動作ではない。

### 2. コード生成による静的解決（`slime export:*` / `slime migrate`）

「命名規則に従ったら自動でコードが生成される」はランタイムではなく**ビルドタイム**で解決する：

```bash
slime export:schema   # parse.tsの型定義 → フロントエンド向け型（静的生成）
slime export:openapi  # route定義 + parse → OpenAPI spec（静的生成）
slime migrate --to-level 6  # 命名規則・ファイル構造を解析 → importパス修正（静的変換）
```

生成された成果物は**人間が読めるテキストファイル**として存在する。魔法はない。何が生成されたかはgit diffで確認できる。

「ファイルを置いたら自動で動く」ではなく「コマンドを叩いたら明示的に生成される」。一手間増えるが、その代わりに追跡可能性が担保される。

### 3. 型システムによる契約表現

「命名規則で暗黙の契約」ではなく、**型で明示的な契約**を表現する：

```ts
// Lv9のPort型: 契約を型で表現、命名規則で強制しない
type ReadPort = {
  findUserById: (id: string) => Promise<User | null>
}

type WritePort = {
  saveUser: (user: User) => Promise<void>
}

// WorkflowはPort型を受け取る（型が合わなければコンパイルエラー）
const createUserWorkflow = async (
  valid: CreateUserInput,
  readPort: ReadPort,
  writePort: WritePort
) => { ... }
```

「このPortに必要なメソッドを実装していれば自動で動く」——でもその検証はコンパイラが行う。ランタイムに初めて気づくのではなく、型が合わなければビルドが落ちる。

## Slime内の「命名規則→自動動作」の整理

Slime/Kaachan設計のなかで「命名規則」が登場する場面と、その解決方法の対応表：

| 命名規則 | 解決方法 | ランタイムか |
|---|---|---|
| Store: `find*/list*/save*/create*` | Kaachan静的検査 | ❌ Lintタイム |
| Logic: ドメインprefixをつける | Kaachan静的検査（prefix混在検出） | ❌ Lintタイム |
| ファイル名: `workflow.ts`, `store.ts` 等 | Kaachan静的検査 + `slime level:current`解析 | ❌ Lintタイム |
| parse.tsの型定義 → フロント型 | `slime export:schema`（明示コマンド） | ❌ ビルドタイム |
| Lv構造 → OpenAPI spec | `slime export:openapi`（明示コマンド） | ❌ ビルドタイム |
| Port型の実装 | TypeScript型検査 | ❌ コンパイルタイム |

**全て静的・明示的。ランタイムに命名規則を「解釈」する仕組みはない。**

## 設計原則のまとめ

### やること

- **命名規則はある**：一貫した命名規則を設け、DXとコードの読みやすさを担保する
- **検査はLintタイム**：命名規則の遵守はKaachanが静的に検査する
- **生成はビルドタイム**：「ファイルから何かを生成する」はコマンドで明示的に行う
- **契約は型で**：暗黙の命名規約ではなく型システムで表現できるものは型にする

### やらないこと

- **ランタイムリフレクション**：命名規則を実行時に解析して動作を変える
- **暗黙のDI/自動登録**：「ここに置いたら自動でコンテナに登録される」系の魔法
- **動的ルーティング**：ファイル名からルートを実行時に解決する（ビルドタイム生成は許容）

### 判断基準

> 「ある命名規則に従えば自動で○○する」設計が浮かんだとき——
> その「自動で○○する」がLintタイムかビルドタイムで解決できるなら**実装する**。
> ランタイムに命名規則を解析する必要があるなら**設計を見直す**。

## 補足：Kaachanとの役割分担

この方針はKaachanの設計哲学とも一致する：

> *「静的解析でできる限界まで踏み込む姿勢がKaachanらしさ」*（FW文書より）

KaachanはAST解析・importグラフ解析・型依存グラフ構築を用いて、命名規則違反・依存方向違反・肥大化を**コードを動かさずに**検出する。これは「ランタイムに命名規則を解決する」の正反対であり、**「コードを実行しなくても、命名規則が守られているかどうかがわかる」**状態を目指している。

LaravelがリフレクションでDXを実現したとすれば、Slime/KaachanはLintとコード生成でDXを実現する。
